2. Task 5.1 — Supabase / Postgres Migration
2.1 Overview
SQLite is replaced by Supabase (managed Postgres) as the production database. All existing SQLAlchemy models remain structurally identical — only the connection driver and string change. Two new tables are added in Phase 5: schwab_tokens and iv_history.
SQLite file is retained locally as a backup and continues to be used for local development. Do not delete signal_matrix.db until Supabase has been running stably in production for at least two weeks.

2.2 New Dependencies
PackagePurposeasyncpgAsync Postgres driver — replaces aiosqlitealembicDatabase migration tool — generates and runs schema migration scriptsschwab-pyOfficial Python wrapper for Schwab Trader API (pip install schwab-py)cryptographyFernet symmetric encryption for token storage at rest
2.3 Connection Strings
Two connection strings are required. Both are saved in .env locally and as Fly.io secrets in production.

Direct connection (Alembic migrations only — never used by the running app):
SUPABASE_CONNECTION_STRING=postgresql+asyncpg://postgres:[PASSWORD]@db.wxqioudsteiwaazrgbao.supabase.co:5432/postgres?ssl=require

Transaction pooler (app runtime — Fly.io and local dev against Supabase):
SUPABASE_POOLED_CONNECTION_STRING=postgresql+asyncpg://postgres.wxqioudsteiwaazrgbao:[PASSWORD]@aws-0-us-east-1.pooler.supabase.com:6543/postgres?ssl=require

backend/database.py reads SUPABASE_POOLED_CONNECTION_STRING at runtime. SUPABASE_CONNECTION_STRING is only used by Alembic CLI commands.

2.4 Migration Strategy
• Run alembic init alembic to initialize migration tooling in backend/
• Configure alembic.ini to use SUPABASE_CONNECTION_STRING (direct, not pooled)
• Run alembic revision --autogenerate -m 'initial schema' to generate migration from existing models
• Review generated migration script before applying
• Run alembic upgrade head against Supabase to create all tables
• Verify all tables exist in Supabase dashboard before proceeding to Task 5.2

4. Task 5.3 — Schwab OAuth Flow
4.1 Overview
schwab-py handles the OAuth 2.0 three-legged flow, token exchange, and token file management. FastAPI wraps schwab-py to expose the auth flow via HTTP endpoints and store tokens encrypted in Supabase. A proactive background refresh task keeps the access token fresh.

4.2 New Files
FilePurposebackend/models/schwab_tokens.pySQLAlchemy model for schwab_tokens tablebackend/services/schwab_client.pyschwab-py client init, token load/save, encryptionbackend/routers/auth.pyOAuth endpoints: login, callback, status, logout
4.3 OAuth Flow
• User visits signal.suttonmc.com, sees red SCHWAB indicator in header
• User clicks indicator ? navigates to GET /api/auth/schwab/login
• FastAPI redirects browser to Schwab authorization URL via schwab-py
• User logs in with Schwab brokerage credentials on Schwab's site
• Schwab redirects to https://signal.suttonmc.com/api/auth/schwab/callback?code=XXXX
• FastAPI receives code, schwab-py exchanges for access + refresh tokens
• Tokens encrypted with Fernet and stored in schwab_tokens table in Supabase
• SCHWAB indicator turns green — app is authenticated
This flow only needs to happen once. The proactive refresh task handles all subsequent token renewal automatically.

4.4 FastAPI Endpoints
EndpointBehaviorGET /api/auth/schwab/loginRedirects browser to Schwab OAuth authorization URLGET /api/auth/schwab/callbackReceives auth code, exchanges for tokens, stores encrypted in SupabaseGET /api/auth/schwab/statusReturns connected/disconnected + token age in secondsDELETE /api/auth/schwab/logoutClears tokens from Supabase — requires re-authentication
4.5 Proactive Token Refresh
Access tokens expire after ~30 minutes. A background task in FastAPI lifespan fires every 25 minutes to refresh proactively — ensuring the 4:00 PM ET data fetch never encounters an expired token mid-run.
# In main.py lifespan context manager (alongside existing APScheduler)
scheduler.add_job(refresh_schwab_token, 'interval', minutes=25)

Refresh logic in schwab_client.py:
• Load tokens from Supabase, decrypt
• Call schwab-py refresh — returns new access token + new refresh token (rolling window)
• Encrypt and write updated tokens back to Supabase
• On refresh failure: mark token as expired, log error, SCHWAB indicator turns red
• Fallback: all Schwab data calls fall back to Yahoo Finance automatically

4.6 Token Expiry Edge Case
If the app is down for more than 7 days (e.g., Fly.io maintenance + extended downtime), the refresh token expires and cannot be renewed automatically. Recovery:
• SCHWAB indicator turns red on next page load
• All data falls back to Yahoo Finance automatically — dashboard remains functional
• User clicks red SCHWAB indicator to re-authenticate
• OAuth flow runs once — normal operation resumes

4.7 Dashboard — SCHWAB Header Indicator
Small status dot added to dashboard header alongside existing SCHED and VIX indicators:
StateDisplayConditionConnected? SCHWAB (green)Token valid, last refresh < 25 min agoAging? SCHWAB (amber)Token valid but refresh token > 6 days oldDisconnected? SCHWAB (red, clickable)Token expired or missing — click to re-auth


5. Task 5.4 — Schwab Quote Polling
5.1 Overview
schwab-py replaces Yahoo Finance as the primary EOD data source. The EOD scheduler gains a new step at 4:00 PM ET for Schwab data fetch, 15 minutes before the existing signal calculation at 4:15 PM ET. Yahoo Finance remains as a silent fallback — it is never removed.

5.2 New File
backend/services/schwab_market_data.py
Contains all Schwab data fetch logic using schwab-py. Yahoo Finance functions in yahoo_finance.py remain untouched — called only as fallback.

5.3 Updated EOD Scheduler Flow
4:00 PM ET — schwab_fetch_all()
    schwab_fetch_quotes()          1 request, all 51 tickers
    schwab_fetch_history()         51 requests, ~26 seconds with 0.5s delay
    writes to price_cache          data_source = 'schwab'

    on SchwabTokenExpiredError:
        yahoo_fetch()              data_source = 'yahoo_fallback'
    on SchwabAPIError:
        yahoo_fetch()              data_source = 'yahoo_fallback'

4:15 PM ET — calculate_signals()   unchanged — reads from price_cache

5.4 Schwab API Endpoints Used
EndpointDataCalls per EODmarketdata/v1/quotesLast price, volume for all tickers1 (batch)marketdata/v1/pricehistoryDaily OHLCV — 60 days per ticker51 (one per ticker)
5.5 Rate Limit Safety
• Schwab limit: 120 requests per minute
• Total EOD requests: 52 (1 batch quote + 51 history calls)
• 0.5 second delay between history calls — completes in ~26 seconds
• Well under 120/min limit — no throttling risk

6. Task 5.5 — IV Percentile
6.1 Overview
Real implied volatility from Schwab options chains replaces the Yahoo Finance realized volatility proxy. IV Percentile is computed as the rank of today's IV within the rolling 252-trading-day history stored in iv_history. Yahoo Rel IV proxy remains as fallback when Schwab is unavailable.

6.2 New File
backend/services/schwab_options.py
Fetches ATM implied volatility from Schwab options chains for each IV-eligible ticker. Writes daily IV to iv_history and computes IV Percentile for price_cache.rel_iv.

6.3 IV-Eligible Tickers
All Tier 1 tickers with standard options chains (~47). Excluded:
TickerReason for ExclusionVIXVolatility index — options exist but not tradeable in standard way$DJIDow Jones index — no standard options chainSPXIndex options (different chain structure from SPY)NDXIndex options (different chain structure from QQQ)
6.4 Schwab API Endpoint
GET /marketdata/v1/chains
  symbol=SPY
  contractType=ALL
  includeUnderlyingQuote=false
  strikeCount=1          # ATM only — minimize response size
  expMonth=ALL

Extract: response['volatility'] — the ATM implied volatility for the front-month contract. This is the single number written to iv_history.implied_vol per ticker per day.

6.5 IV Percentile Calculation
# For each IV-eligible ticker at EOD:
today_iv = schwab_options_chain(ticker)['volatility']

# Fetch last 252 trading days from iv_history
history = db.query(IVHistory)
    .filter(IVHistory.ticker == ticker)
    .order_by(IVHistory.iv_date.desc())
    .limit(252).all()

# Rank today's IV within history (0-100)
from scipy.stats import percentileofscore
iv_values = [row.implied_vol for row in history]
iv_percentile = percentileofscore(iv_values, today_iv, kind='rank')

# Write to price_cache
cache_row.rel_iv = iv_percentile
cache_row.iv_source = 'schwab'

6.6 Bootstrap Period
On day one, iv_history is empty. IV Percentile becomes more meaningful as history accumulates. Cold start approach: use Yahoo Finance Rel IV proxy tagged as 'proxy' until sufficient Schwab IV history exists.
History AvailableIV SourceAccuracy0-20 daysYahoo proxy (fallback)Approximate — realized vol proxy21-63 daysSchwab IV (partial history)Directionally correct64-252 daysSchwab IV (growing history)Increasingly accurate252+ daysSchwab IV (full 52-week)Full accuracy — true IV Percentile

4:00 PM ET

    schwab_fetch_iv()              NEW — 47 options chain calls
        ~0.5s delay between calls  ~24 seconds total

        writes price_cache.rel_iv + iv_source

7. Task 5.6 — OBV Source Swap
7.1 Overview
The OBV pivot engine in conviction_engine.py reads volume_history_json from price_cache. This field is populated by yahoo_finance.py today and will be populated by schwab_market_data.py in Phase 5. The OBV engine itself requires zero changes — it is source-agnostic by design.

7.2 Change Required
In yahoo_finance.py, the PHASE 5 TODO comment marks the exact swap point:
# PHASE 5 TODO: Replace this block with Schwab volume history
# volume_history_json is populated here from hist['Volume']
# After swap: schwab_market_data.py populates this field instead
# conviction_engine.py OBV engine reads volume_history_json regardless of source

After swap: schwab_market_data.py populates volume_history_json from candles[].volume in the Schwab pricehistory response. Same JSON array format, same date alignment as history_json. The yahoo_finance.py block is commented out (not deleted) as reference.

7.3 Volume Quality Improvement
Schwab pricehistory returns true exchange volume — the same number visible on ThinkorSwim. Yahoo Finance volume can differ due to after-hours and pre-market aggregation methodology. Schwab volume is cleaner input for OBV pivot detection.

7.4 Phase 5 OBV Upgrade — Volume Surge (Future)
Deferred to Phase 6 but noted here for continuity. When Schwab streaming is live:
• Swap volume_history_json source from Schwab REST to Schwab streaming
• Add volume surge icon to dashboard row (150%+ of 20-day avg)
• Real-time intraday volume makes OBV signal more current and accurate
